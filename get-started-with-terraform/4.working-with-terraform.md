# Working with Terraform

### 1. Terraform configuration syntax 
- **Hashicorp Configuration Language (HCL)** is the terraform syntax `Native Terraform Language` + `JSON`.
- The configuration files extension is `.tf`
- comfiguration files are composed of `Blocks` and `Arguments`
- Each block has a `type`(A number of `labels`):\
**aws**, **aws_vpc** and **main_vpc** are all labels.\
```tf
provider "aws"{
    ...
}


resource "aws_vpc" "main_vpc"{
    ...
}
```
- some blocks require "NO Label"\
example: **required_providers**
```tf
terraform{
    required_providers{
        aws = {
            source = "hashicorp/aws"
            version = "~> 3.0"
        }
    }
}
```
- some blocks require "One Label"\
example: **aws**
```tf
provider "aws"{
    ...
}
```
- some blocks require "Two Labels"\
example: **aws_vpc** and **main_vpc**
```tf
resource "aws_vpc" "main_vpc"{
    ...
}
```

- Inside Blocks, we find *arguments = value* and *nested blocks*
- Comments in HCL:
```tf
# One line comment AND end of line comment
// One line comment

/*
Multi lines
comment
*/
```

### 2. Working directory
#### a. terraform init
- Go to the main directory ou working directory (The one containing the `main.tf` file)
- Run `terraform init`: this will search for terraform providers in the registry, download and install them locally.
- This command is **Idempotent**
```bash
$ terraform init
Initializing the backend...
Initializing provider plugins...
- Finding hashicorp/aws versions matching "~> 6.0"...
- Installing hashicorp/aws v6.41.0...
- Installed hashicorp/aws v6.41.0 (signed by HashiCorp)
Terraform has created a lock file .terraform.lock.hcl to record the provider
selections it made above. Include this file in your version control repository
so that Terraform can guarantee to make the same selections by default when
you run "terraform init" in the future.

Terraform has been successfully initialized!

You may now begin working with Terraform. Try running "terraform plan" to see
any changes that are required for your infrastructure. All Terraform commands
should now work.

If you ever set or change modules or backend configuration for Terraform,
rerun this command to reinitialize your working directory. If you forget, other
commands will detect it and remind you to do so if necessary.
```
- Now Terraform can perform any operation like **provisionning infrastructures** or **Modifying its state**

#### b. Terraform plan
- run `terraform plan` to preview the change it will be making
- to save those changes to run them later, run `terraform plan -out=tfplan1
- This will create a plan file called tfplan1
- later execute that by running `terraform apply tfplan1`

#### c. Terraform Apply
- run `terraform apply` to deploy those changes
- run `terraform apply -auto-approve` to avoid confirmation
- Run `terraform apply tfplan1` to run the plan saved in the **tfplan1** file.
- Run `terraform apply -replace="Resource_type.Resource_name"` to force resource replacement(Destroy & Create).\
Ex: If something changes inside the EC2 Instance, terraform will not be avare of that. `terraform apply` will not trigger any change. Use `terraform apply -replace="Resource_type.Resource_name"` to force replacement.

#### d. Terraform Format
- To correct the configurations files format in the current  directory, run: `terraform fmt`.
- To correct formating in config files of a specific directory, run :  `terraform fmt directory_name`.
- To correct the format and view what has changed, run : `terraform fmt -diff`
- To view what formatting will change before applying, run : `terraform fmt -diff -check`
- To correct formatting in config files the current directory and subdirectories, run: `terraform fmt -recursive`

#### e. Terraform Validate
- `terraform validate` Validates the syntax in terraform config files.
- Also knows about blocks and arguments. 
- Shows the line of the syntax error
- Gitve the error type (`Error: Unsupported argument`)
- Can make a proposition of the correct syntax

#### f. Terraform Destroy
There are many way to destroy reseources with terraform
- Destroy ALL ressources in the config files: `terraform destroy -auto-approve`
- Destroy some resources specified: `terraform destroy -target resource_type.resource_name -auto-approve` \
EX: `terraform destroy -target aws_vpc.myvpc`
>[!NOTE]
>
> - You can delete this way only if the resource is managed by terraform.
> - It is not a good practice to destroy specific resources this way.
> - Use it for particular situations only.

- Destroy some resources by delete them or comment them from the config files and run `terraform apply` (BEST WAY to keep things consistent)

### 3. `.terraform.lock.hcl` File
- `.terraform.lock.hcl` file is created by  **terraform init**
- It is the dependency lock file that get created or updated any time you run **terraform init**
- It captures the versions of all terraaform providers that you are using

>[!NOTE]
>
>`.terraform.lock.hcl` Should be added to the version control, so that team members that run terraform get the exact same versions.

### 4. `.terraform` Directory
- `.terraform` file is created by  **terraform init**
- It is a local cache, where terraform stores some files needed for subsequent operations agains to the same configuration
- For exemple. il could contain the downloaded providers

>[!NOTE]
>
>`.terraform` directory Should NOT be added to the version control.
