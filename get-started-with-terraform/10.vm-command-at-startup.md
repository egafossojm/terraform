# How to run commands on VM at startup

### 1. user_data
This is the recommended way.

- The user_data attribute will paste the commands to the cloud provider which runs them on the instance at startup.
- This is idiomatic(Best Practice) and native to the cloud provider.
- User data in viewable in the AWS console
- You can add user data `inline` or from a `script`.

EX1: Inline
```tf
resource "aws_instance" "my_vm" {
  ami = data.aws_ami.amazon_linux_2023.image_id
  instance_type = "t2.micro"
  vpc_security_group_ids = [aws_default_security_group.default_sg.id]
  associate_public_ip_address = true
  key_name = aws_key_pair.test_ssh_key.key_name 
  subnet_id = aws_subnet.web.id
  user_data = <<EOF
#!/bin/bash
sudo yum -y update && sudo yum -y install httpd
sudo systemctl start httpd && sudo systemctl enable httpd
sudo bash -c 'echo "<h1> Deployed via Terraform</h1>" > /var/www/html/index.html'
EOF 
  tags = {
    "Name" = "${var.main_vpc_name} VM"
  }
}
```

Ex2: Script file
```bash
$cat entry-script.sh 
#!/bin/bash
sudo yum -y update
sudo yum -y install httpd
sudo systemctl start httpd
sudo systemctl enable httpd
sudo bash -c 'echo "<h1> Deployed via Terraform</h1>" > /var/www/html/index.html'
```
```tf
resource "aws_instance" "my_vm" {
  ami = data.aws_ami.amazon_linux_2023.image_id
  instance_type = "t2.micro"
  vpc_security_group_ids = [aws_default_security_group.default_sg.id]
  associate_public_ip_address = true
  key_name = aws_key_pair.test_ssh_key.key_name 
  subnet_id = aws_subnet.web.id
  user_data = file("entry-script.sh")
  tags = {
    "Name" = "${var.main_vpc_name} VM"
  } 
}
```

### 2. cloud-init
This is the **industry standard method for cloud instance initialisation** : https://cloud-init.io/

- `cloud-init` is The standard for customizing cloud instances.
- It Works with many popular operating systems
- It is supported accros all major cloud providers, private cloud providers** and bare-metal installations.
- cloud-init can run per instance or per boot configuration.
- **First boot**: Should run `per instance` configuration.
- **Subsequent boot**: Should run `per boot` configuration.

Ex:

- Create the instance template (`instance-config.yaml`):
```yaml

  - devops: [root,sys] # Group devops with members(root,sys)
  - hashicorp          # Empty group

# Adding users to the system. Users are added after groups are added
users:
  - default
  - name: terraform
    gecos: terraform
    shell: /bin/bash
    primary_group: hashicorp
    sudo: ALL=(ALL) NOPASSWD:ALL
    groups: users, admin
    lock_passwd: false
    ssh_authorized_keys:
      - ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQCvX4gLfvIgOR1iO/yTUiDOuVt1YCV29HivKic5iXWxejZDejFnZk8chDnelqkZC8TkPympfb2se5pzPhdY1W8q+Ml8NA5UmMUJ2XQ/K4yel+TVygvm8qLmB6vnVxKzbstrmHaEGLwWBMnBKI+O52aI2e+IGT7+mhJHY3v3Wzd0nLpAmogJTzmvqyaifpEeqrpfBz9JQCGwXK7FPR9kT6wS7I8SZfoLdF+4bggKQPWvKmV3zlgg4i+TW+s6GjHDgr1t46J/fOksKfkN3qa9rKu99biWXLXcXmr58YjuyJBTJfnCyuiVCGnb15hsBDixhGZKhXhBwZq5lrDKYWh443kN testing-ssh-key

# Downloading and installing packages

  - sudo docker run -d -p 8080:80 nginx
```

- In the `main.tf` add a **template_file** data source with the cloud-init template:
```tf
data "template_file" "user_data" {
  template = file("instance-config.yaml")
}
```

- Then provide the template_file data source to instance user data.
```tf
resource "aws_instance" "my_vm" {
  ami                         = data.aws_ami.amazon_linux_2023.image_id
  instance_type               = "t2.micro"
  vpc_security_group_ids      = [aws_default_security_group.default_sg.id]
  associate_public_ip_address = true
  key_name                    = aws_key_pair.test_ssh_key.key_name
  subnet_id                   = aws_subnet.web.id
  user_data = data.template_file.user_data.rendered
  tags = {
    "Name" = "${var.main_vpc_name} VM"
  }
}
```

>[!NOTE]
>
>```tf
> user_data = data.template_file.user_data.rendered
>```
> `.rendered` replaces variables in the template_file data source(if any) and provides text to user_data.


### 3. terraform provisioners

>[!NOTE]
>
>`user_data`
> - It is native to the cloud provider.
> - It is the recommended way to customize instances.
>
>`provisioners`
> - They native to Terraform.
> - They are a least resort and should be used with care.
> - Breaks the declarative model of terraform
> - Adds complexity
> - Terraform will connect it self to the instance (using ssh in most cases), and run commands on the system.

EX:

- Add `connection` block to establish connection to the instance.
```tf
  connection {
    type = "ssh"
    host = self.public_ip
    user = "ec2-user"
    private_key = file("~/.ssh/test_key")
  }
  ```
  - Add a `file provisioner` to copy the script on the instance
  ```tf
    provisioner "file" {
    source = "./entry-script.sh"
    destination = "/home/ec2-user/entry-script.sh"
  }
  ```
- Add `remote-exec provisioner` to execute commands on the instance.
```
  provisioner "remote-exec" {
    inline = [
      "chmod +x /home/ec2-user/entry-script.sh",
      "/home/ec2-user/entry-script.sh",
      "exit"
     ]
     on_failure = continue
  }
  ```

>[!NOTE]
>
> If you not specify `on_failure = continue`, terraform will stop in case of error during commands execution.

- Complete exemple:


```
resource "aws_instance" "my_vm" {
  ami                         = data.aws_ami.amazon_linux_2023.image_id
  instance_type               = "t2.micro"
  vpc_security_group_ids      = [aws_default_security_group.default_sg.id]
  associate_public_ip_address = true
  key_name                    = aws_key_pair.test_ssh_key.key_name
  subnet_id                   = aws_subnet.web.id

  connection {
    type = "ssh"
    host = self.public_ip
    user = "ec2-user"
    private_key = file("~/.ssh/test_key")
  }

  provisioner "file" {
    source = "./entry-script.sh"
    destination = "/home/ec2-user/entry-script.sh"
  }

  provisioner "remote-exec" {
    inline = [
      "chmod +x /home/ec2-user/entry-script.sh",
      "/home/ec2-user/entry-script.sh",
      "exit"
     ]
     on_failure = continue
  }

  tags = {
    "Name" = "${var.main_vpc_name} VM"
  }

}
```
